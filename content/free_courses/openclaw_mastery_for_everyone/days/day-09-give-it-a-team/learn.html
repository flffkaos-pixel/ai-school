9일차: 팀을 구성하세요

오늘 배울 내용:

  • 하위 에이전트 작동 방식: 자동(OpenClaw가 생성) 및 수동(모델과 사고 수준을 선택하여 직접 생성)
  • 실행 중인 하위 에이전트와 실시간으로 상호 작용하는 방법: 진행 상황 관찰, 후속 메시지 보내기, 작업 중간 조정
  • 명명된 에이전트를 통해 하나의 게이트웨이에서 각각 고유한 작업 공간, ID 및 성격을 가진 여러 특수 "브레인"을 실행할 수 있는 방법
  • 상담원이 동의 메시지를 통해 서로 통신하는 방법

오늘 만들 항목: 오늘이 끝날 때까지 Claw에는 자신만의 개성과 목소리를 지닌 전문 작가인 팀원이 있습니다.

시작하기 전 참고 사항: 오늘의 기사는 과정에서 가장 긴 기사입니다.

하나의 발톱에서 여러 발톱까지

지난 8일 동안 귀하는 하나의 Claw를 만드셨습니다.

다중 에이전트 설정에 대한 YouTube 동영상을 본 적이 있다면 여러 에이전트가 함께 작업하고 각각이 더 큰 작업을 처리하거나 각각 별도의 도메인을 소유하는 등 다른 점을 보셨을 것입니다.

이제 하나의 Claw가 할 수 있는 모든 것을 이해했으므로 팀을 구성할 준비가 되었습니다.

발톱에 팀을 부여하는 두 가지 방법

하위 에이전트는 임시 직원입니다.

명명된 에이전트는 끈질긴 두뇌입니다.

이렇게 생각해보세요. 하위 에이전트는 특정 작업을 위해 고용된 임시 계약자입니다.

하위 에이전트: 자동 모드

하위 에이전트가 작동하는 가장 일반적인 방법은 자동입니다.

하나의 오케스트레이터 에이전트가 작업을 수신하고, 이를 독립적인 하위 작업으로 나누고, 명확한 지침으로 작업자 에이전트를 생성하고, 해당 출력을 수집하고, 최종 결과를 종합합니다.

각 작업자는 격리된 에이전트 인스턴스입니다.

이러한 격리는 의도적인 것이며 품질을 향상시킵니다.

하위 에이전트: 수동 모드

자동 생성은 편리하지만 명시적인 제어가 필요한 경우도 있습니다.

하위 에이전트를 수동으로 생성하면 즉시 백그라운드에서 작업이 시작됩니다.

수동 생성은 주요 대화를 중단하지 않고 주요 대화와 함께 실행하려는 경우에 유용합니다.

실행 중인 하위 에이전트와 상호 작용

하위 에이전트는 실행되는 동안 완전히 대화형으로 작동합니다.

모든 활성 하위 에이전트를 나열하고, 그 중 하나에 대한 자세한 정보를 보거나, 활동 로그를 읽어 그들이 호출한 도구와 얻은 결과를 정확하게 확인할 수 있습니다.

가장 유용한 명령은 초점입니다.

이러한 수준의 제어를 통해 하위 에이전트는 팀을 감독하는 것처럼 느껴집니다.

비용 라우팅: 저렴한 인력, 유능한 오케스트레이터

하위 에이전트는 비용 라우팅을 실용적으로 만듭니다.

"500 단어로 된 이 기사를 읽고 주요 주장을 추출"하는 작업을 수행하는 작업자는 빠르고 저렴한 모델에서 이를 잘 처리합니다.

하위 작업 분류를 설계하고, 작업자 출력의 품질을 평가하고, 최종 결과를 종합하는 오케스트레이터는 보다 유능한 모델의 이점을 얻습니다.

구성의 모든 하위 에이전트에 대한 기본 모델을 설정하고 필요할 경우 생성별로 이를 재정의합니다.

하위 에이전트 가드레일

하위 에이전트는 강력하므로 가드레일이 필요합니다.

하위 에이전트가 자동으로 종료되기 전까지 실행할 수 있는 시간에 대한 시간 초과가 있습니다.

하위 에이전트가 액세스할 수 있는 도구를 제한할 수도 있습니다.

하위 에이전트를 사용해야 하는 경우

하위 에이전트에 적합한 사례는 다음과 같은 작업입니다.

완전히 평행합니다. 다섯 가지 독립적인 연구 질문입니다.

컨텍스트 격리. 때로는 한 작업의 출력이 다른 작업의 컨텍스트와 별도로 유지되어야 합니다.

시간에 민감합니다. 여러 주제에 대한 조사가 포함된 아침 요약을 예약하는 경우 병렬 작업자는 순차 처리보다 훨씬 빠르게 완료할 수 있습니다.

하위 에이전트는 오버헤드를 추가합니다.

자동 대 수동: 작업이 간단할 때("5개 회사 조사") Claw가 자동으로 생성되도록 하세요.

명명된 에이전트: 여러 두뇌, 하나의 게이트웨이

하위 에이전트는 단일 대화 내에서 병렬 작업을 처리합니다.

명명된 에이전트는 완전히 격리된 두뇌입니다.

이를 통해 VPS에서 하나의 게이트웨이를 실행하고 여러 전문 에이전트를 호스팅할 수 있습니다.

네임드 에이전트의 진정한 힘은 제너럴리스트와 스페셜리스트를 짝을 이룰 때 나타납니다.

이 과정에서는 설정이 OpenClaw 채팅 내부에 유지됩니다.

에이전트 간 통신

명명된 에이전트는 서로 완전히 격리되기 시작합니다.

연결해야 할 이유가 있으면 에이전트 간 메시징을 활성화하고 통신이 허용되는 에이전트를 지정합니다.

패턴은 옵트인(opt-in), 에이전트별, 명시적입니다.

언제 어느 것을 사용해야 하는가?

다음과 같은 경우 하위 에이전트를 사용하세요.

  • 독립적인 작업을 병렬로 실행해야 하는 경우
  • 근로자는 임시적이고 일회용이다.
  • 비용 라우팅 문제(저렴한 인력, 유능한 코디네이터)
  • 특정 모델에 대해 일회성 작업을 수동으로 생성하려고 합니다.

다음과 같은 경우 지정 에이전트를 사용하세요.

  • 도메인(직장, 개인, 가족) 간의 지속적인 분리를 원합니다.
  • 작업에는 전문적인 성격과 SOUL.md의 혜택을 받는 심층적인 도메인 전문 지식이 필요합니다.
  • 다양한 채널이 다양한 상담원에게 도달해야 합니다.
  • 특정 상황에서 사용할 수 있는 도구를 제한하고 싶습니다.

둘 다 함께 사용하세요. 작업 에이전트는 병렬 연구 작업을 위해 하위 에이전트를 생성할 수 있습니다.

시간을 투자하세요. 이제 다중 에이전트 설정을 이해할 수 있는 충분한 컨텍스트가 있으므로 9일차에 이를 다루겠습니다.

먼저 하나의 발톱에 익숙해지세요.

오늘의 빌드에서는 패턴의 구체적인 예로 전문가 한 명을 생성합니다.

구축할 준비가 되셨나요?

이제 Claw에 팀을 구성하는 두 가지 방법을 이해하게 되었습니다.

내일은 마지막 날입니다. 귀하가 구축한 모든 것에 대한 전체 검토, 10일 전체에 대한 검증 및 과정 평가입니다.

더 깊게

  • 채널 라우팅을 사용하면 명명된 에이전트를 특정 채널이나 대화에 바인딩할 수 있습니다.
  • 오케스트레이터/작업자 패턴에는 더 복잡한 변형이 있습니다. 즉, 계획 에이전트가 작업을 중단하고 전문 작업자가 병렬로 실행하며 합성 에이전트가 결과를 결합하는 3계층 계층 구조입니다.
  • 하위 에이전트 오류 처리에 의존하기 전에 먼저 이해해 볼 가치가 있습니다.
  • OpenClaw 커뮤니티는 복잡한 결정을 위해 함께 호출할 수 있는 전문 에이전트(재무 분석용, 기술 검토용, 법적 고려용)라는 "협의회" 설정을 위한 패턴을 개발했습니다.
  • 에이전트별 샌드박스는 공유 장치에 대해 살펴볼 가치가 있습니다.
  • "다중 에이전트 LLM 시스템이 실패하는 이유"는 다중 에이전트 실패 모드에 대한 가장 포괄적인 연구입니다.
