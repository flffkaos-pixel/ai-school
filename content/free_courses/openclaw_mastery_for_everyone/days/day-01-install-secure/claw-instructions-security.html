1일차: 보안 확인

이 파일에는 Hostinger의 새로운 OpenClaw 배포에 대한 보안 검사가 포함되어 있습니다.

각 섹션을 순서대로 실행하세요.

위험 상태에 대한 메시지가 표시되면 VPS 강화를 선택합니다. 기본적으로 거부, 최소 표면, 가장 엄격한 설정입니다.

1. OS 정보

운영 체제, 커널 버전 및 프로세스가 루트가 아닌 사용자로 실행 중인지 여부를 보고합니다.

예상: Debian 12, 13, Ubuntu 22.04/24.04.

지원되고 패치된 OS가 기반입니다.

2. 포트 열기

바인딩된 주소와 함께 현재 수신 중인 모든 포트를 나열합니다.

ss -tlnp

  • 127.0.0.1(OpenClaw 게이트웨이)의 포트 18789
  • 127.0.0.1에 몇 가지 추가 OpenClaw 내부 포트
  • 일반적으로 60000+ 범위의 0.0.0.0에 포트 1개

해당 0.0.0.0 포트가 Control UI입니다.

게이트웨이 자체는 절대로 0.0.0.0에 있어서는 안됩니다.

3. 방화벽

컨테이너 내부에 방화벽이 활성화되어 있는지 확인하세요.

예상: 방화벽 도구가 없습니다(UFW, iptables 또는 nft 없음).

베어 VPS에서는 방화벽을 활성화해야 합니다.

4. OpenClaw 보안 감사

기본 제공 보안 감사를 실행합니다.

openclaw security audit --deep

  • 역방향 프록시 헤더가 신뢰할 수 없음(신뢰할 수 있는 프록시가 설정되지 않음)
  • 확장 플러그인에 대한 허용 도구 정책
  • 고정되지 않은 플러그인 npm 버전

이 단계에서는 개인 용도로 사용할 위험이 낮으며 정보 제공용입니다.

5. 게이트웨이 구성

~/.openclaw/openclaw.json을 읽고 다음 값을 확인하세요.

기대값
게이트웨이 모드 local
게이트웨이 주소 127.0.0.1
인증 모드 token
dm정책 제한적인 값으로 설정하거나 설정하지 않습니다.
그룹 정책 disabled으로 설정되거나 설정되지 않습니다.

토큰 인증은 모든 요청에 ​​유효한 토큰이 필요함을 의미합니다.

잘못된 값이 있으면 openclaw.json에서 수정하세요.

6. 파일 권한

자격 증명 디렉터리에 대한 권한을 확인합니다.

ls -ld ~/.openclaw/credentials


chmod 700 ~/.openclaw/credentials

7. 채널

현재 활성화된 메시징 채널과 저장된 자격 증명이 있는지 여부를 나열합니다.

예상: 여러 채널(Telegram, WhatsApp, Discord, Slack 등)이 활성화되어 있지만 구성되지 않은 것으로 표시될 수 있습니다.

채널에 이미 자격 증명이 있는 경우 해당 자격 증명이 어디서 왔는지 보고하세요.

8. 웹 검색

웹 검색이 활성화되어 있는지 확인하세요.

예상: 이 단계 후에는 비활성화됩니다.

10. 재시작 및 최종 확인


게이트웨이를 다시 시작하고 두 검사를 한 번 더 실행합니다.


openclaw gateway restart
openclaw doctor
openclaw security audit
