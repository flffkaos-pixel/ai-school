4부: 검색 증강 생성(RAG) 및 에이전트 RAG의 등장

이전 부분에서는 AI 에이전트가 이메일 전송, 티켓 파일 전송, API 트리거 등 실제 시스템과 상호 작용하는 데 도구가 어떻게 도움이 되는지 살펴보았습니다.

하지만 모델이 조치를 취해야 합니까?
올바른 정보에 액세스해야 한다면 어떻게 될까요?

그게?셲 많은 기업 환경의 경우입니다.

  • 내부 문서가 팀 전체에 분산됨
  • 아무도 쓴 것을 기억하지 못하는 정책 PDF
  • CRM 노트에 묻혀 있는 고객 통찰력
  • 유용한 컨텍스트가 포함된 대시보드 및 이메일

도구가 이겼나요?셳 여기서 도와주세요.

RAG 란 무엇입니까?

RAG는 ​​Retrieval-Augmented Generation을 의미합니다.
그것은 응답을 생성하기 직전에 모델이 자신의 데이터에서 관련 정보를 검색하는 시스템 설계입니다.

RAG는 ​​모델이 훈련된 내용에만 의존하는 대신 기업 시스템의 실시간 상황 정보에 대한 액세스를 제공합니다.

당신은 궁금할 것입니다:

?쏻모든 데이터를 모델에 직접 제공하는 것은 아닌가요???/p>

  • 모델은 한 번에 제한된 양의 텍스트만 처리할 수 있습니다.
  • 그 한도 내에서도 관련성이 없거나 시끄러운 정보가 너무 많이 포함되면 어려움을 겪습니다.
  • 이로 인해 응답의 집중도가 떨어지고 오류가 발생하기 쉽습니다.

RAG 프로세스(한눈에 보기)

여기요?셲 실제 모습은 다음과 같습니다.

  1. 데이터 ??내부 콘텐츠(PDF, 이메일, 메모, 위키)
  2. 청킹 ??더 나은 색인 생성을 위해 더 작은 부분으로 나뉩니다.
  3. 프롬프트 + 컨텍스트 ??쿼리 시 시스템이 관련 부분을 검색합니다(검색 단계).
  4. LLM ??모델은 해당 컨텍스트를 사용하여 응답을 생성합니다.
  5. 출력 ??결과는 모델 ?쐊현재??/li>가 아닌 데이터를 기반으로 합니다.
  6. 모델 미세 조정은 비용이 많이 들고 느립니다.
  7. 검색이 더 빠르고 안전하며 제어하기 쉽습니다.
  8. LLM 시스템에 구조와 추적성을 제공합니다.
  9. 비정형(문서) 데이터와 반정형(대시보드, 메모) 데이터 모두에서 작동합니다.

    10. 따라서 질문하는 대신:

    ?쏦우리가 아는 모든 것을 모델에게 가르쳐야 할까요???br>대부분의 팀에서는 다음과 같이 묻습니다.
    ?쏦이제 우리가 이미 가지고 있는 것을 모델이 가져오도록 해야 합니까???/p>

    RAG의 핵심은 간단합니다.

    • LLM을 수강하세요
    • 생성 직전에 검색된 추가 정보를 제공합니다.

    이를 통해 모델이 더 정확하고 상황을 더 잘 인식하며 기억된 사실에 덜 의존하게 됩니다.
    이 기능은 특히 Q&A, 요약, 정책 조회와 같은 작업에 유용하며 특히 법률, 재무, 지원과 같은 데이터가 풍부한 환경에서 더욱 유용합니다.

    2024년이 ?쐔RAG의 해라고 불리는 것도 당연합니다.??/strong>

    하지만 이제 우리는 에이전트 시대로 접어들고 있습니다.

    RAG는 ​​사라지고 있지만 진화하고 있습니다.

    오늘날의 시스템은 그렇지 않습니다.
    한 번만 검색하면 답변이 생성됩니다.
    에이전트 워크플로에서 검색은 더 광범위하고 동적 추론 루프의 일부가 됩니다.

    에이전트는 한 번이 아니라 작업 전반에 걸쳐 필요한 만큼 여러 번 계획하고 검색하고 반영하고 다시 검색합니다.

    그게?셲 Agentic RAG가 들어오는 곳이죠.

    Agentic RAG란 무엇입니까?

    전통적인 RAG:

    • 하나의 쿼리
    • 한 번의 검색
    • 하나의 응답

    다음과 같은 독립형 질문에 적합합니다.

    ?쏻모?셲 PTO 롤오버에 대한 당사 정책???/p>

    예:
    예를 들어 영업팀을 위한 거래 도우미를 구축한다고 가정해 보겠습니다.
    단일 작업에서 상담원은 다음을 수행해야 할 수 있습니다.

    • 고객을 끌어당긴다?셲 CRM 이력
    • 해당 세그먼트의 현재 가격 검색
    • 지역별 법률 용어를 찾아보세요
    • 과거 계약 조항 참조
    • 맞춤형 제안 생성
    • 사실을 다시 확인하세요
    • 상호작용 기록

    에이전트 시스템에서 검색은 ?셳 단지 설정 단계일 뿐입니다.
    ?셲 에이전트가 수행하는 방법은 다음과 같습니다.

    • 누락된 컨텍스트 수집
    • 가정을 확인합니다.
    • 작업 중간에 적응

    이는 RAG가 다음과 같이 됨을 의미합니다.

    • 과제 내 학습을 위한 도구
    • 환각을 줄이는 방법
    • 동적 작업흐름을 처리하기 위한 메커니즘
    • 추론과 기초적인 기업 지식 사이의 연결

    Agentic RAG는 검색을 모델 사고 과정의 일부로 사용하여 검색을 최고 수준의 의사 결정 루프로 전환합니다.

    도구로서의 RAG

    생각해 보면 RAG도 일종의 도구입니다.
    그러나 작업을 실행하는 대신 에이전트가 대용량 데이터에서 올바른 정보를 가져오는 데 도움이 됩니다.

    실제로 상담원은 다음을 결합하는 경우가 많습니다.

    • 래그
    • 도구
    • 계획

    ?쫡복잡한 작업을 안정적이고 상황에 맞게 완료합니다.

    범위에 대한 참고 사항

    RAG는 ​​깊고 빠르게 발전하는 공간입니다. 솔직히 말해서 그 자체가 하나의 과정일 수도 있습니다.
    더 자세히 알아보고 싶다면 다음을 참조하세요.

    • 나는 풍경을 잘 다루는 주요 RAG 논문의 GitHub 저장소를 선별했습니다.
    • Agentic RAG에 대한 101 가이드도 있습니다

    하지만 모든 사용 사례에 모든 RAG 최적화가 필요한 것은 아닙니다.
    6주 과정에서는 각 기술을 맹목적으로 적용하기보다는 각 기술이 언제 어디서 적합한지 이해하는 데 중점을 둡니다.

    다음 부분에서는 최근 가장 많이 회자되는 개념 중 하나인 모델 컨텍스트 프로토콜(MCP)에 대해 살펴보겠습니다.

    최대한 활용하려면 도구에 관한 3부를 다시 방문하는 것이 좋습니다. MCP는 해당 개념을 직접 기반으로 구축되기 때문입니다!
