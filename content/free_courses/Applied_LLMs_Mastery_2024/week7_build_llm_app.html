이 가이드는 간단한 적용을 통해 기본 요소를 익히는 것을 목표로 기본 사항을 다루도록 설계되었습니다.

각 섹션을 탐색하면서 구성 요소로 앱을 빌드하는 데 도움이 되는 몇 가지 리소스를 선택하고 계속 진행하세요.

2. 프롬프트 체인 연결(프롬프트 체인 + LLM)


프롬프트 템플릿을 활용하고 LLM을 호출하는 것이 효과적이지만 때로는 다음 질문을 하기 위해 이전에 얻은 답변을 사용하여 LLM에 여러 질문을 차례로 질문해야 할 수도 있습니다. 


LangChain은 LLM 응용 프로그램 작성을 위해 널리 사용되는 라이브러리로 등장하여 LLM을 통해 여러 질문과 답변을 연결하여 단일 최종 응답을 생성할 수 있습니다. 


prompt1 ="{질문}에 관한 다음 질문은 어떤 주제입니까?"



RAG(Retrieval-Augmented Generation)는 LLM에게 답변하기 전에 확인할 개인 라이브러리를 제공하는 것과 같습니다. 


LLM 및 프롬프트를 넘어서 RAG(Retrieval-Augmented Generation)를 구현하려면 다음과 같은 기술 요소가 필요합니다.


지식 기반, 특히 벡터 데이터베이스


시스템이 정보를 찾기 위해 활용할 수 있는 문서, 기사 또는 데이터 항목의 포괄적인 모음입니다. 


벡터화된 지식 기반을 갖는 목적은 효율적인 유사성 검색을 가능하게 하는 것입니다. 


검색 구성요소


검색 구성 요소는 사용자의 쿼리와 관련된 정보를 찾기 위해 지식 기반의 실제 검색을 수행하는 엔진입니다. 


    

  1. 쿼리 인코딩: 지식 베이스를 벡터화하는 데 사용된 것과 동일한 모델이나 방법을 사용하여 사용자의 쿼리를 벡터로 변환합니다. 
    2. 

  2. 유사성 검색: 쿼리가 벡터화되면 검색 구성요소는 벡터 데이터베이스에서 가장 가까운 벡터를 검색합니다. 
    3. 

  3. 정보 검색: 가장 가까운 벡터를 식별한 후 검색 구성요소는 지식 베이스에서 해당 항목을 가져옵니다. 
    4. 

  4. 집계(선택사항): 일부 구현에서는 검색 구성요소가 여러 소스의 정보를 집계하거나 요약하여 통합 응답을 제공할 수도 있습니다. 
    5. 



RAG 프레임워크에서는 검색 구성 요소의 출력(즉, 검색된 정보)이 원본 쿼리와 함께 LLM에 제공됩니다. 


By combining a vectorized knowledge base with an efficient retrieval mechanism, RAG systems can provide answers that are both highly relevant and deeply informed by a wide array of sources. This approach is particularly useful in applications requiring up-to-date information, domain-specific knowledge, or detailed explanations that go beyond the pre-existing knowledge of an LLM.


LangChain과 같은 프레임워크는 이미 RAG 프레임워크를 구축하기 위한 좋은 추상화를 갖추고 있습니다.

We've explored chaining and incorporating knowledge. Now, consider the scenario where we need to remember past interactions in lengthy conversations with the LLM, where previous dialogues play a role.


This is where the concept of Memory comes into play as a vital component. Memory mechanisms, such as those available on platforms like LangChain, enable the storage of conversation history. For example, LangChain's ConversationBufferMemory feature allows for the preservation of messages, which can then be retrieved and used as context in subsequent interactions. You can discover more about these memory abstractions and their applications on LangChain's 선적 서류 비치.

이전 섹션에서는 도구 및 메모리와 같은 복잡한 LLM 구성 요소를 살펴보았습니다. 


LLM 에이전트는 바로 이 작업을 수행하며 LLM을 계획, 메모리 및 도구 사용과 같은 다른 모듈과 결합하여 복잡한 작업을 수행하도록 설계된 시스템입니다. 


예를 들어, LLM 에이전트가 재무 계획을 지원하기를 원하는 시나리오를 생각해 보세요. 


이 작업을 수행하기 위해 에이전트는 먼저 메모리 모듈을 활용하여 개인의 지출, 수입원 및 재정적 목표에 관해 저장된 데이터에 액세스합니다. 


    

  1. 데이터 분석: 에이전트는 외부 도구를 사용하여 재무 데이터를 처리하고, 비용을 분류하고, 추세를 파악하고, 총 지출, 저축률, 지출 분포와 같은 주요 측정항목을 계산합니다.
    2. 

  2. 예산 평가: LLM 에이전트는 분석된 데이터를 기반으로 개인의 재무 목표를 달성하는 데 있어 현재 예산의 효율성을 평가합니다. 
    3. 

  3. 추천 생성: 에이전트는 재무 원칙과 최적화 전략에 대한 이해를 활용하여 개인의 재무 건전성을 개선하기 위한 맞춤 추천을 작성합니다. 
    4. 

  4. 커뮤니케이션: 마지막으로 LLM 에이전트는 자연어 생성 기능을 사용하여 각 제안의 근거와 잠재적 이점을 설명함으로써 명확하고 이해하기 쉬운 방식으로 사용자에게 권장 사항을 전달합니다.
    5. 



이 프로세스 전반에 걸쳐 LLM 에이전트는 의사 결정 능력을 외부 도구, 메모리 저장소 및 계획 메커니즘과 원활하게 통합하여 사용자의 재정 상황에 맞는 실행 가능한 통찰력을 제공합니다.


Here's how LLM agents combine various components to make decisions:


    

  1. 언어 모델(LLM): LLM은 에이전트의 중앙 컨트롤러 또는 '브레인' 역할을 합니다. 
    2. 

  2. 주요 모듈:
      


    • 메모리: 메모리 모듈을 사용하면 에이전트가 현재 작업과 관련된 정보를 저장하고 검색할 수 있습니다. 
      • 

    • 도구 사용: 에이전트는 외부 도구나 API를 활용하여 데이터를 수집하고 계산을 수행하거나 출력을 생성할 수 있습니다. 
      • 

    

    3. 



기존 프레임워크는 에이전트 구성을 위한 내장 모듈과 추상화를 제공합니다.
