[4주차] 검색 증강 생성

ETMI5: 5에서 설명해주세요

이번 주 콘텐츠에서는 응답 생성 프로세스 중에 외부 소스에서 실시간으로 상황에 맞는 관련 정보를 통합하여 대규모 언어 모델의 기능을 향상시키는 AI 프레임워크인 검색 증강 생성(RAG)에 대해 심층적으로 살펴보겠습니다.

RAG는 ​​수집, 검색 및 합성의 세 가지 주요 단계로 작동합니다.

RAG의 역사를 논의한 다음 수집, 검색, 합성을 포함한 RAG의 주요 구성 요소를 자세히 살펴보고 각 단계의 프로세스와 개선 전략에 대한 자세한 통찰력을 제공합니다.

RAG란 무엇인가요?

RAG(Retrieval Augmented Generation)는 생성 프로세스 중에 외부 소스로부터 상황에 맞는 최신 정보를 통합하여 LLM에서 생성된 응답의 품질을 향상시키는 AI 프레임워크입니다.

역사

RAG(Retrieval-Augmented Generation)는 Meta의 이 논문에서 첫선을 보였습니다. 

Image Source: [https://arxiv.org/pdf/2005.11401.pdf](https://arxiv.org/pdf/2005.11401.pdf)

RAG는 ​​빅 언어 모델이 사실을 기억하고 특정 작업을 수행하는 데는 능숙하지만 해당 지식을 정확하게 사용하고 조작하는 데 어려움을 겪었기 때문에 탄생했습니다.

그런 다음 검색 증강 생성을 위한 유연한 미세 조정 방법 형태의 게임 체인저인 RAG가 등장했습니다.

RAG의 주요 의미는 시스템에 비모수적 메모리 추가를 제안했던 과거 접근 방식에서 벗어난 것입니다.

주요 구성 요소

앞서 언급했듯이 RAG의 핵심 요소에는 섭취, 검색 및 합성 프로세스가 포함됩니다.

음식물 섭취

RAG에서 수집 프로세스는 응답 생성을 위해 모델에서 데이터를 활용하기 전에 데이터를 처리하고 준비하는 것을 의미합니다.

이 프로세스에는 3가지 주요 단계가 포함됩니다.

  1. 청킹: B입력 텍스트를 더 작고 관리하기 쉬운 세그먼트 또는 청크로 다시 정리합니다.
  2. 삽입: 텍스트나 덩어리를 계산 친화적인 방식으로 필수 품질을 포착하는 벡터 형식으로 변환합니다.
  3. 색인 생성: 빠르고 효율적인 검색을 위해 최적화된 구조화된 형식으로 삽입된 데이터를 구성합니다.

검색

검색 구성 요소에는 다음 단계가 포함됩니다.

  1. 사용자 쿼리: 사용자가 LLM에 자연어 쿼리를 제출합니다.
  2. 벡터 비교: 쿼리의 숫자 벡터는 이전 단계에서 생성된 지식 기반의 인덱스에 있는 벡터와 비교됩니다.
  3. Top-K 검색: 그런 다음 시스템은 쿼리 벡터와 가장 유사한 지식 기반에서 상위 K 문서 또는 구절을 검색합니다.
  4. 데이터 검색: 시스템은 지식 베이스에서 선택된 상위 K 문서에서 실제 콘텐츠 또는 데이터를 검색합니다.

따라서 검색 단계가 끝나면 LLM은 사용자 쿼리와 가장 관련성이 높은 지식 기반 세그먼트와 관련된 관련 컨텍스트에 액세스할 수 있습니다.

합성

합성 단계는 이제 LLM이 지식 기반의 추가 컨텍스트에 액세스할 수 있다는 점을 제외하면 일반 LLM 생성과 매우 유사합니다.

RAG 챌린지

RAG는 ​​LLM과 지식을 통합하는 매우 간단한 방법인 것처럼 보이지만 아래에 언급된 RAG에 대한 공개 연구 및 적용 문제는 여전히 남아 있습니다.

  1. 데이터 수집 복잡성: 광범위한 지식 기반을 수집하는 복잡성을 처리하려면 엔지니어링 문제를 극복해야 합니다.
  2. 효율적인 임베딩: 대규모 데이터세트를 효율적으로 임베딩하려면 속도 제한 해결, 강력한 재시도 논리 구현, 자체 호스팅 모델 관리와 같은 문제가 발생합니다.
  3. 벡터 데이터베이스 고려 사항: 벡터 데이터베이스에 데이터를 저장하려면 컴퓨팅 리소스 이해, 모니터링, 분할 및 잠재적인 병목 현상 해결과 같은 고려 사항이 필요합니다.
  4. 미세 조정 및 일반화: 다양한 지식 집약적 NLP 작업 전반에 걸쳐 일반화를 보장하면서 특정 작업에 맞게 RAG 모델을 미세 조정하는 것은 어렵습니다.
  5. 하이브리드 파라메트릭 및 비파라메트릭 메모리:RAG와 같은 모델에 파라메트릭 및 비파라메트릭 메모리 구성 요소를 통합하면 지식 수정, 해석 가능성 및 환각 방지와 관련된 문제가 발생합니다.
  6. 지식 업데이트 메커니즘: 실제 지식이 발전함에 따라 비모수적 메모리를 업데이트하는 메커니즘을 개발하는 것이 중요합니다.

RAG 구성요소 개선(수집)

1. 더 나은 청킹 전략

RAG 구성 요소에 대한 수집 프로세스를 향상시키는 맥락에서 텍스트 데이터를 효율적으로 처리하려면 고급 청킹 전략을 채택하는 것이 필요합니다.

대규모 데이터 세트의 복잡성을 고려하여 최근에는 다음 전략이 사용되고 있습니다.

  1. 콘텐츠 기반 청킹: 품사 태깅이나 구문 분석과 같은 기술을 사용하여 의미와 문장 구조를 기반으로 텍스트를 분류합니다.
  2. 문장 청킹: 문장 경계 인식 또는 음성 세그먼트를 사용하여 텍스트를 완전하고 문법적으로 올바른 문장으로 나누는 작업이 포함됩니다.
  3. 재귀적 청킹: 텍스트를 다양한 수준의 청크로 분할하여 계층적이고 유연한 구조를 만듭니다.

2.

향상된 색인 기능을 통해 정보를 보다 효율적으로 검색할 수 있습니다.

  1. 상세 색인 생성: 하위 부분(예: 문장)을 통해 청크를 만들고 각 청크의 위치를 ​​기반으로 식별자를 할당하고 콘텐츠를 기반으로 특징 벡터를 할당합니다.
  2. 질문 기반 색인 생성: 지식 영역(예: 주제)을 통해 청크를 분류하고 각 청크에 카테고리를 기반으로 식별자를 할당하고 관련성을 기반으로 특성 벡터를 할당합니다.
  3. 청크 요약으로 최적화된 색인 생성: 추출 또는 압축 기술을 사용하여 각 청크에 대한 요약을 생성합니다.

4. 쿼리 변환 및 라우팅 [소스]

쿼리 변환 방법은 복잡한 쿼리를 하위 질문으로 나누고(확장) 재작성을 통해 잘못 표현된 쿼리를 개선하여 검색을 향상시킵니다.

쿼리 변환

  1. 쿼리 확장: 쿼리 확장은 입력을 하위 질문으로 분해하며, 각 하위 질문은 더 좁은 검색 문제입니다.
  2. 쿼리 재작성: 프레임이 잘못되었거나 단어로 작성된 사용자 쿼리를 처리하는 다시 쓰기-검색-읽기 접근 방식에는 검색 효과를 높이기 위해 질문을 다시 작성하는 작업이 포함됩니다.
  3. 쿼리 압축: 사용자 질문이 광범위한 채팅 대화 뒤에 이어지는 시나리오에서는 질문에 대답하기 위해 전체 대화 컨텍스트가 필요할 수 있습니다.

쿼리 라우팅

  1. 동적 쿼리 라우팅: 데이터가 어디에 있는지에 대한 질문은 RAG, 특히 다양한 데이터 저장소가 있는 프로덕션 설정에서 매우 중요합니다.

RAG 구성 요소 개선(세대)

LLM 생성을 위한 가장 간단한 방법은 모든 관련 컨텍스트 조각을 연결하고 사전 정의된 관련성 임계값을 초과하며 이를 쿼리와 함께 단일 인스턴스에서 LLM에 제공하는 것입니다.

1. 응답 종합 접근법:

3단계가 필요합니다.

  1. 반복적 구체화: 검색된 컨텍스트를 언어 모델에 덩어리 단위로 보내 답변을 구체화합니다.
  2. 요약: 프롬프트에 맞게 검색된 컨텍스트를 요약하고 간결한 답변을 생성합니다.
  3. 다중 답변 및 연결: 다양한 문맥 덩어리를 기반으로 여러 답변을 생성한 다음 이를 연결하거나 요약합니다.

2. 인코더 및 LLM 미세 조정:

이 접근 방식에는 RAG 파이프라인 내에서 LLM 모델을 미세 조정하는 작업이 포함됩니다.

  1. 인코더 미세 조정: 더 나은 임베딩 품질과 컨텍스트 검색을 위해 Transformer 인코더를 미세 조정합니다.
  2. 랭커 미세 조정: 특히 기본 인코더에 대한 신뢰가 부족한 경우 검색된 결과의 순위를 다시 매기려면 크로스 인코더를 사용하세요.
  3. RA-DIT 기술: RA-DIT와 같은 기술을 사용하여 쿼리, 컨텍스트 및 답변의 세 가지 요소에 대해 LLM과 Retriever를 모두 조정합니다.
